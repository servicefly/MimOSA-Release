## 1. Summary

M4.1 fulfils the Phase 4 promise (reserved by the `SkillsSettings.custom` slot
in Phase 3) of letting users teach MimOSA **new commands without writing any
Python**. A custom skill is a small, declarative specification — trigger
phrases, a match mode, and a response — stored in the user's local config.
Custom skills are loaded into the live intent router and matched **ahead of** the
generic question/LLM fallback, so user intent always wins.

Crucially, custom skills are **data, not code**: there is no `eval`, no
`exec`, no shell-out. A custom skill can only emit a fixed text response or ask
the configured LLM a templated question. This keeps the feature safe and
privacy-preserving by construction.

## 2. What was delivered

| Capability | Detail |
|------------|--------|
| Declarative spec | `CustomSkillSpec` dataclass — id, name, triggers, match mode, response type/value, enabled flag |
| Match modes | `any` (default), `all`, `exact`, `regex` |
| Response types | `text` (fixed reply) and `llm` (templated prompt to the configured provider) |
| Safe by design | No code execution; regex is compiled defensively and errors are contained |
| Router integration | New **Tier 1c** custom-match stage between greeting patterns and the question-shape heuristic |
| Config integration | `SkillsSettings` validates, de-dups & normalizes custom entries; add/remove helpers |
| Live loading | `VoiceLoop.router` builds custom skills from config on construction |

### New / changed modules

| File | Change |

## 4. Privacy & safety notes

- Custom skills never execute arbitrary code.
- `llm`-type responses honour the global provider setting, including
  `provider = none` (fully offline) — in which case the skill degrades to a
  friendly "LLM unavailable" reply rather than failing.
- All specs live only in the local config file; nothing is uploaded.
